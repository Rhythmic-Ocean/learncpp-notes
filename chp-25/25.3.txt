25.3 — The override and final specifiers, and covariant return types
  -override specifiers:
    we had one problem with virtual functions, if they had different return types/ parameters
    between the base and derived, then the virtual keyword in derived will be invalid/ now work as 
    expected, to fix that we can use the override keyword

    eg:
      class A
      {
      public:
        virtual std::string_view getName1(int x) { return "A"; }
        virtual std::string_view getName2(int x) { return "A"; }
      };

      class B : public A
      {
      public:
        virtual std::string_view getName1(short x) { return "B"; } // note: parameter is a short
        virtual std::string_view getName2(int x) const { return "B"; } // note: function is const
      };

      int main()
      {
        B b{};
        A& rBase{ b };
        std::cout << rBase.getName1(1) << '\n';
        std::cout << rBase.getName2(2) << '\n';

        return 0;
      }

    Here: A is printed twice

  -To protect from the uninteded output bit, we use override specifier that way invalid overrides
    live above will result in direct compile time error
      class A
      {
      public:
        virtual std::string_view getName1(int x) { return "A"; }
        virtual std::string_view getName2(int x) { return "A"; }
      };

      class B : public A
      {
      public:
        virtual std::string_view getName1(short x) override { return "B"; } // note: parameter is a short
        virtual std::string_view getName2(int x) const override { return "B"; } // note: function is const
      };

      int main()
      {
        B b{};
        A& rBase{ b };
        std::cout << rBase.getName1(1) << '\n';
        std::cout << rBase.getName2(2) << '\n';

        return 0;
      }
    now compile time error in both case
  NOTE: If we have both const and override, then const must be listed first

-If we don't want a function/ class to be oeverided, we use final:
  In case of virtual functions:
    class A{
    public:
      virtual std::string_view getName() const {return "A";} 
    };

    class B: class A{
    public:
      std::string_view getName() const override final {return "B";}
    }

    clas C: class B{
    public:
      std::string_view getName() const override {return "C";} -> ERROR
    }
  To prevent class inheritence we use it after class name:
      class A
      {
      public:
        virtual std::string_view getName() const { return "A"; }
      };

      class B final : public A // note use of final specifier here
      {
      public:
        std::string_view getName() const override { return "B"; }
      };

      class C : public B // compile error: cannot inherit from final class
      {
      public:
        std::string_view getName() const override { return "C"; }
      };

Case where different return types is okay for virtual override, usually if return type of a virtual
  function is pointer or reference to some class, the override function can return pointer or a 
  reference to a derived class of that class. This is called covariant return types
  eg:
  class Base{
  public:
    virtual Base* getThis() {std::cout << "called Base::getThis()"; return this;}
    void printType() {std::cout << "Base";}
  };
  class Derived{
    virtual Derived* getThis() override {std::cout << "called Derived::getThis()"; return this;}
    void printType() {std::cout << "Derived";}
  }
  int main(){
    Derived d {};
    Base * b{&d};
    d.getThis().printType(); -> prints "called Deriged::getThis()" and "Derived"
    b->getThis().printType(); -> prints "called Derived::getThis()" and "Base"
  }

  The second case cuz override is successful so Deriged::getThis() is called, but since we are
    calling thru Base*, return this will return Base* so next call to printType() will be to 
    Base::printType()
