  Default constructor and stuff:
    -constructors can be overloaded like normal functions
    -empty user defined constructor: constructor with no argument that does not assign to/ initialize any members
      class Foo{
        int m_x {1};
        int m_y {2};
        Foo(){ -> 
          std::cout << m_x << m_y;
        }
      }
    -explicitly defaulted default constructor;
      class Foo{
        int m_x {1};
        int m_y {2};
        Foo() = default;
      }
    -implicitly defaulted default constructor;
      class Foo{
        int m_x {1};
        int m_y {2};
        //implicitly defauted default constructor;
      }
  NOTE: both second and last cases above used to be considered aggregrates, now only the last one is considered aggregrate
  NOTE: in case of both 
    -implicitly defaulted default constructor AND
    -explicitly defaulted default constructor:
      the member variables are default/ zero initialized 
    But with empty user defined constructor, the members are left as it is, and if the member is unitialized, then it will just have garbage value and will be left unitialized

    Example:

      class User
      {
      private:
          int m_a; // note: no default initialization value
          int m_b {};

      public:
          User() {} // user-defined empty constructor

          int a() const { return m_a; }
          int b() const { return m_b; }
      };

      class Default
      {
      private:
          int m_a; // note: no default initialization value
          int m_b {};

      public:
          Default() = default; // explicitly defaulted default constructor

        int a() const { return m_a; }
        int b() const { return m_b; }
    };

    class Implicit
    {
    private:
        int m_a; // note: no default initialization value
        int m_b {};

    public:
        // implicit default constructor

        int a() const { return m_a; }
        int b() const { return m_b; }
  };

  int main()
  {
      User user{}; // default initialized
      std::cout << user.a() << ' ' << user.b() << '\n';

      Default def{}; // zero initialized, then default initialized
      std::cout << def.a() << ' ' << def.b() << '\n';

      Implicit imp{}; // zero initialized, then default initialized
      std::cout << imp.a() << ' ' << imp.b() << '\n';

      return 0;
  }

  -Explicilty default constructor is useful when u wanna have an option for your class to be default/ zero intialized too:

  Copy Constructors:
    -when we do:
      Frac fra {1, 2};
      Fraf f {fra}; 
      We are basically initializing a copy constructor and f will now have a shallow copy of fra (or whatever's in the stack, if it's just int then all of it)

    -copy constructor are like this:
      class Frac{
        int m_num {};
        int m_deno {};
      public: 
        Frac(const Frac& f) : m_num {f.m_num}, m_deno {f.m_deno}{ -> overriding the copy constructor ourselves!
          std::cout << "Copy created!!";
        }
      }

    -whenever we pass an object by value we are basically calling a copy constructor. 
      eg: let's take the class right above this:

        Frac generateFrac(int a, int b){
          Frac f {a, b};
          return f;
        }

        void print(Frac f){...}

        int main(){
          Frac f {generateFrac(1, 2)};
          print(f);
        }
      In this case the copy constructor will go off three times, first when value is returned form generateFrac, second when that value is copied to f and finally when fi s passed 
        to print()

    -generate default copy constructor:
      Frac(const Frac& f) = default;
    -disable/ delete copy constructor:
      Frac(const Frac& f) = delete;
  
  NOTE: r value can only be assigned to const reference, cuz const reference extents it's lifetime. Non const objects does not have that effect!!
  NOTE: for copy constructor we pass by reference not value cuz we need copy constructor to copy an object, so having to call a function while defining it will just make infinite
        chains of calls

14.15
  -copy elison:
    When you call a copy constructor to create an object, the compiler can optimize away the argument to the copy constructor and directly create the object.
      eg:
        Frac f {Frac {1, 2}};
        could be direct optimizaed to:
          Frac f{1, 2};
      if it happens we say the parameter to copy constructor is elided away!

    NOTE: this can cause changes in observable program effect so it's highly discouraged to modify the copy constructor and introduce some kind of side effect in it!

14.17
  Constexpr aggregrates and class:
    -A member function can be constexpr but for it to be evaluated at compile time we also need the corresponding object to be constexpr. If we tried assigning an expression with
      constexpr member function but non constexpr object then compile time error
    struct Pair{
        int m_x {};
        int m_y {};
      
        constexpr int sum() const{
            return m_x + m_y
        }
      }

      int main(){
        Pair p {1, 2};
        std::cout << p.sum();  -> works
        constexpr int g {p.sum()}; -> compile time error! Obj's NOT constexpr
        constexpr Pair q {1, 2};
        constexpr int m = {q.sum()}; -> works!
      }
    NOTE: The above thingy only works for aggregrates! For classes doing as above will give compile type error saying q is not a literal type to be a constexpr. What's a literal is
        is a little complex, but for userdefiend classes it must either be an aggregrate or have a constexpr constructor
          eg:
        class Pair // Pair is no longer an aggregate
        {
        private:
            int m_x {};
            int m_y {};

        public:
            Pair(int x, int y): m_x { x }, m_y { y } {}

            constexpr int greater() const
            {
                return (m_x > m_y  ? m_x : m_y);
            }
        };

        int main()
        {
            constexpr Pair p { 5, 6 };       // compile error: p is not a literal type
            std::cout << p.greater() << '\n';

            constexpr int g { p.greater() };
            std::cout << g << '\n';

            return 0;
        }
    

    -A brief category of literal types:
      Scalar types (those holding a single value, such as fundamental types and pointers)
      Reference types
      Most aggregates
      Classes that have a constexpr constructor

    -The fix for above is just to make the constructor a constexpr:
        #include <iostream>

        class Pair
        {
        private:
            int m_x {};
            int m_y {};

        public:
            constexpr Pair(int x, int y): m_x { x }, m_y { y } {} // now constexpr

            constexpr int greater() const
            {
                return (m_x > m_y  ? m_x : m_y);
            }
        };

        int main()
        {
            constexpr Pair p { 5, 6 };
            std::cout << p.greater() << '\n';

            constexpr int g { p.greater() };
            std::cout << g << '\n';

            return 0;
        }

    NOTE: implicitly defaulted default constructor are constexpr by default, but for explicitly defaulted constructor they must be constexpr explicitly!
      constexpr Foo() = default;
    NOTE: When a constexpr function is evaluating in a compile-time context, only constexpr functions can be called.
    NOTE: In C++11 contexpr member functions were implicitly const by default i.e you couldn't change "this". But stating C++14, that's no longer the case.
    -So starting C++14:
          constexpr int greater() const // constexpr and const
          {
              return (m_x > m_y  ? m_x : m_y);
          }

          constexpr void reset() // constexpr but non-const
          {
              m_x = m_y = 0; // non-const member function can change members
          }

    NOTE: A really really weird case:
      constexpr Pair zero()
      {
          Pair p { 1, 2 }; // p is non-const
          p.reset();       // okay to call non-const member function on non-const object
          return p;
      }
      Here, p and p.reset will be computed at compile time as long as Pair's constructor and Pair::reset() both are of constexpr type!
      So when u call zero onto a constexpr fnc (forcing it at compile time), it can go and do it!
    NOTE: A constexpr expression does not need to be initialized with a literal or a const value, what we need is a constant value i.e a value that can be generated at compile time
      so even tho zero() is NOT a const, we can still do:
        constexpr Pair q {zero()}; ->(this will prob prompt the compiler to do some copy elision and elide away the unnecessary stuff)

    -constexpr const member function returning const pointer that's a const:
